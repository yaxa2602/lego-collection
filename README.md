# 🧱 Коллекция LEGO

Каталог всех наборов LEGO и личная коллекция: отмечайте, что у вас есть и что хотите, смотрите статистику по франшизам и годам, делитесь коллекцией по ссылке.

**Живая версия: [lego-collection-plum.vercel.app](https://lego-collection-plum.vercel.app)**

Каталог — более 27 000 наборов с 1949 года, ~150 франшиз. Данные берутся из Rebrickable API и кэшируются на стороне сервера.

---

## Возможности

### Каталог (`/`)
- Поиск по названию или номеру набора.
- Фильтр по франшизе — **поле с поиском** по 150 темам (клавиатура: стрелки, Enter, Escape), при выборе франшизы мгновенно подставляются её подколлекции.
- Фильтр **«только в продаже»** — отсекает наборы, которые, скорее всего, уже сняты с производства.
- Сортировка: по году (в обе стороны), названию, числу деталей.
- Карточки в стиле магазина: при наведении выезжают кнопки **«Есть»** и **«Хочу»** (на тач-экранах видны сразу), в углу — метка ✓ / ★, если набор уже в коллекции.
- Пагинация с номерами страниц и полем «перейти к №».

### Страница набора (`/set/[номер]`)
- Фото, тема (полный путь «франшиза → подколлекция»), год, число деталей и минифигурок.
- Бейдж доступности: **«В продаже»** или **«Снят с производства»**.
- Блок «Где купить» — поиск набора на Avito, Ozon, eBay, Amazon и BrickLink; отдельно — официальная инструкция по сборке с lego.com.
- **«Соберите серию»** — другие наборы той же темы, которых у вас ещё нет.

### Моя коллекция (`/mine`)
- Вкладки «Есть» и «Хочу», группировка по франшизам.
- Статистика: количество наборов, деталей и минифигурок, разбивка по франшизам и годам.
- Кнопка «Поделиться» создаёт публичную ссылку.

### Публичная коллекция (`/c/[slug]`)
Витрина коллекции по секретной ссылке, доступна без регистрации. Показывается только раздел «Есть» — вишлист остаётся приватным.

## Как это устроено

Несколько решений, которые определили архитектуру:

**Кэш каталога.** У Rebrickable лимит около одного запроса в секунду, поэтому наборы и дерево тем кэшируются в Postgres (`sets_cache`, `themes_cache`) с TTL в 30 дней. Запись в кэш идёт «мимо» пользователя: если она упадёт, ответ всё равно вернётся. Если сам API недоступен, страница набора отдаёт данные из кэша.

**Фильтры считает API, а не приложение.** «Только в продаже» превращается в параметр `min_year` запроса к Rebrickable. Если бы фильтрация шла на нашей стороне, счётчик и пагинация показывали бы неправду («найдено 27 350», а на странице три карточки).

**Безопасность на уровне базы.** Row Level Security включён на всех таблицах: коллекцию видит и меняет только её владелец. Клиент работает с публичным ключом, и даже подделанный запрос не даст доступа к чужим данным. Сервисный ключ используется только на сервере, для записи в кэш.

**Шаринг без утечек.** Публичная страница не обращается к таблице `shares` напрямую — она закрыта политикой. Чтение идёт через `security definer`-функцию, которая по секретному слагу отдаёт только раздел «Есть». Слаг — 12 символов из `crypto.randomUUID()`.

**Рендеринг.** Каталог, страница набора и коллекция — серверные компоненты; фильтры работают обычной GET-формой. Клиентский JavaScript только там, где он действительно нужен: кнопки коллекции, поле поиска франшиз, всплывающие подсказки.

**Единый источник ссылок.** Все площадки описаны в одном месте (`lib/links.ts`), поэтому страница набора, модальное окно и уведомление физически не могут разойтись. Порог «снят с производства» тоже один (`lib/availability.ts`) — им пользуются и бейдж, и фильтр каталога.

## Стек

- [Next.js 16](https://nextjs.org) — App Router, TypeScript, серверные компоненты
- [Supabase](https://supabase.com) — Postgres с RLS, аутентификация по email
- [Rebrickable API](https://rebrickable.com/api/) — данные о наборах
- Обычный CSS, без UI-фреймворков
- [Vitest](https://vitest.dev) — тесты доменной логики
- Деплой на [Vercel](https://vercel.com), автоматически при пуше в `master`

## Структура

```
app/            страницы: каталог, набор, коллекция, публичная ссылка, вход
components/     карточки, кнопки коллекции, статистика, фильтр тем, уведомления
lib/            Rebrickable + кэш, дерево тем, статистика, ссылки, доступность
supabase/       SQL-миграции (схема, RLS-политики, security definer функция)
```

## Запуск локально

```bash
npm i
cp .env.example .env.local   # и заполнить переменные
npm run dev
```

Приложение поднимется на [http://localhost:3000](http://localhost:3000).

Переменные окружения (`.env.local`):

| Переменная | Назначение |
| --- | --- |
| `NEXT_PUBLIC_SUPABASE_URL` | адрес Supabase-проекта |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | публичный ключ (виден в браузере, доступ ограничен RLS) |
| `SUPABASE_SERVICE_ROLE_KEY` | сервисный ключ, только для серверных операций |
| `REBRICKABLE_API_KEY` | ключ [Rebrickable API](https://rebrickable.com/api/) |

Миграции из `supabase/migrations/` применяются в SQL-редакторе Supabase по порядку.

## Тесты и проверки

```bash
npm test           # unit-тесты (Vitest)
npx tsc --noEmit   # проверка типов
npm run build      # production-сборка
```

Тестами покрыта доменная логика: сборка запросов к API и маппинг ответов, дерево тем и путь «франшиза → подколлекция», агрегация статистики, формирование внешних ссылок, оценка доступности набора.

## Источники данных и ограничения

Честно о том, чего приложение не делает:

- **Цен нет намеренно.** Rebrickable их не отдаёт, а официальные цены (Brickset API) требуют отдельного ключа. Показывать выдуманную оценку «по числу деталей» — врать точностью, которой нет, поэтому вместо неё даются ссылки-поиски по площадкам, где цены живые.
- **Доступность — оценка, а не факт.** Бейдж «снят с производства» вычисляется по году выпуска: LEGO обычно держит набор в рознице два-три года. Точных дат снятия в бесплатных источниках нет.
- **Фильтр по франшизе не разворачивает вложенные темы.** Rebrickable принимает только одно значение `theme_id` за запрос, поэтому выбор франшизы ищет по ней самой, а подколлекция — по подколлекции.
- **Названия наборов на английском** — так они и записаны в базе Rebrickable, это привычно для LEGO-сообщества. Интерфейс при этом русский.

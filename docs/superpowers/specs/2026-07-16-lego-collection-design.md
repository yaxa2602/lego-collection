# Спека: «Коллекция LEGO» — каталог наборов и личная коллекция

Дата: 2026-07-16. Статус: одобрена пользователем (дизайн-диалог в Claude Code).

## Цель

Публичное веб-приложение (PWA-friendly), где любой пользователь ведёт свою коллекцию LEGO:
ищет наборы в полном каталоге, отмечает «у меня есть» / «хочу», смотрит статистику
коллекции и делится ею ссылкой. Второй портфолио-проект пользователя (Кворк);
ключевые новые навыки относительно первого проекта: интеграция с внешним REST API
(Rebrickable) с кэшированием и дашборд со статистикой/графиками.

## Стек

- Next.js (App Router, TypeScript) — как в проекте «Карта уличных животных».
- Supabase: Auth (email+пароль), Postgres (коллекции пользователей + кэш каталога).
- Хостинг: Vercel.
- Данные о наборах: **Rebrickable API v3** (бесплатный ключ). Ключ хранится только на сервере.
- Тесты: Vitest (доменная логика).

## Экраны

### 1. Каталог (`/`)
- Поиск по названию или номеру набора.
- Двухуровневый фильтр по темам Rebrickable: франшиза → подколлекция
  (например, Star Wars → UCS). Дерево тем берём из `/lego/themes/`.
- Сортировка: по году, числу деталей, названию.
- Карточки: фото, название, номер, год, число деталей.
- Пагинация (постранично, как отдаёт API).

### 2. Страница набора (`/set/[setNum]`)
- Большое фото, название, номер, год, тема (полный путь: франшиза → подколлекция),
  число деталей, число минифигурок (эндпоинт `/lego/sets/{set_num}/minifigs/`).
- Кнопки: «У меня есть», «Хочу» (для гостя — ведут на вход).
- Ссылка «Инструкции по сборке» → официальная страница
  `https://www.lego.com/service/building-instructions/<номер без суффикса -1>`.
- Блок «Купить / оценить»: ссылки-поиски по номеру и названию набора:
  - BrickLink: страница набора/поиска по `set_num`;
  - Avito: поиск `lego <номер> <название>`.
- Если получен ключ Brickset API — показываем рекомендованную розничную цену (RRP).
  Без ключа блок цены не показывается (ссылки-поиски остаются). Живые цены вторички
  не парсим и не обещаем.

### 3. Моя коллекция (`/mine`)
- Две вкладки: «Есть» и «Вишлист», внутри — группировка по франшизам.
- Панель статистики: всего наборов, всего деталей, разбивка по темам и по годам
  (простые графики — столбики/донат; без тяжёлых chart-библиотек, SVG своими руками
  или лёгкая библиотека).
- Удаление из коллекции, перенос «Хочу» → «Есть».
- Кнопка «Поделиться» → создаёт/показывает ссылку `/c/[shareSlug]`.

### 4. Публичная коллекция (`/c/[shareSlug]`)
- Read-only витрина чужой коллекции: те же вкладки и статистика, без кнопок правки.
- Доступна без регистрации. Владелец может отключить шаринг (slug удаляется).

### 5. Вход (`/login`)
- Email+пароль через Supabase Auth (тот же паттерн, что в карте животных).

## Данные

### Rebrickable API
- Все запросы к Rebrickable — только с сервера (Route Handlers / Server Components),
  ключ в env `REBRICKABLE_API_KEY`.
- Лимит API ≈ 1 запрос/сек ⇒ обязательный кэш в Postgres:
  - `sets_cache` (set_num PK, name, year, theme_id, num_parts, num_minifigs,
    img_url, fetched_at) — записывается при каждом обращении к набору;
    повторные просмотры и коллекции читаются из кэша. TTL мягкий: данные наборов
    почти не меняются, обновляем при fetched_at старше 30 дней.
  - `themes_cache` (id PK, parent_id, name, fetched_at) — всё дерево тем одним
    прогоном, обновление раз в 30 дней.
- Поиск/каталог идёт через API напрямую (кэшировать все страницы поиска — YAGNI),
  результаты попутно пишутся в `sets_cache`.

### Наши таблицы (Supabase, все с RLS)
- `collection_items`: id, user_id, set_num, status ('owned' | 'wishlist'),
  added_at. Уникальность (user_id, set_num). RLS: читает/пишет только владелец
  (+ публичное чтение через shareSlug — см. ниже).
- `shares`: user_id PK, slug (уникальный, случайный), created_at.
  Публичная страница читает коллекцию через slug: RLS-политика «чтение
  collection_items разрешено, если у владельца есть строка в shares» —
  либо чтение через серверный Route Handler с service-ключом строго по slug
  (решение за этапом плана; предпочтителен вариант без service-ключа).
- Статистика считается на лету из collection_items + sets_cache (наборов у
  человека десятки — агрегация тривиальна).

## Внешний вид

Минимализм, «LEGO-дух дозированно» (подтверждено пользователем):
- светлая тема — основная, тёмная включается по `prefers-color-scheme`;
- нейтральный светлый фон, много воздуха, аккуратная сетка карточек;
- главный акцент — красный LEGO (кнопки, активные фильтры, бейджи статусов);
  жёлтый и синий — вторичные акценты, точечно;
- скругления «под кубик», декоративные студы (шишечки) на панелях/шапке;
- тёмная тема — по `prefers-color-scheme`, без кислотных цветов;
- шрифт с характером, но читабельный (подбор на этапе реализации, навык
  frontend-design).
- Язык интерфейса: русский (данные наборов из API — на английском, это норма
  для LEGO-сообщества). Переключатель RU/EN — вне скоупа v1.

## Ошибки и граничные случаи

- Rebrickable недоступен/лимит: страница набора отдаёт данные из `sets_cache`
  с пометкой «данные из кэша»; поиск показывает понятную ошибку с кнопкой повтора.
- Набор не найден: подсказка формата («номер вида 42151»).
- Пустая коллекция: дружелюбная заглушка с кнопкой «Найти первый набор».
- Slug шаринга не существует/отключён: страница «коллекция недоступна».
- Двойное добавление набора — невозможно (уникальный индекс), кнопка меняет статус.

## Тесты (Vitest)

- Агрегация статистики (наборы/детали/по темам/по годам).
- Построение дерева тем и путь «франшиза → подколлекция».
- Формирование внешних ссылок (lego.com instructions, BrickLink, Avito).
- Маппинг ответов Rebrickable в наши типы.

## Вне скоупа v1

- Живые рыночные цены (парсинг вторички).
- Социальность: профили, лента, поиск людей.
- Учёт отдельных деталей/минифигурок, «что я могу собрать».
- Переключатель языка, push-уведомления.

## Порядок работ (крупно)

1. Скаффолд Next.js + Supabase, auth по образцу карты животных.
2. Обёртка Rebrickable API + кэш-таблицы + миграции.
3. Каталог с поиском/фильтрами/сортировкой.
4. Страница набора (+ внешние ссылки).
5. Коллекция: добавление/статусы/статистика/графики.
6. Шаринг.
7. Визуальная полировка (frontend-design), PWA-мелочи, деплой на Vercel.
